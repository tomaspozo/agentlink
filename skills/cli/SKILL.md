# CLI

The `create-agentlink` CLI scaffolds new Supabase projects and updates existing ones. It handles infrastructure setup, template files, database configuration, and migration generation.

---

## Commands

### Scaffold a new project

```bash
npx create-agentlink <name> [prompt]
```

Interactive wizard that: checks prerequisites, starts Supabase (or creates a cloud project with `--cloud`), sets up the database (extensions, schemas, functions, queues, auth triggers), scaffolds a React + Vite + React Router v7 frontend (default) or Next.js (`--nextjs`), writes template files, generates migrations, configures Claude Code, installs MCP + plugin + companion skills, then launches Claude Code with the prompt.

### Scaffold in an existing project

```bash
cd my-project && npx create-agentlink
```

Detects the existing directory and integrates AgentLink into it. Requires a clean git working tree.

### Update an existing project

```bash
npx create-agentlink --force-update
```

Re-applies template files, patches `config.toml`, runs SQL setup, and regenerates migrations if schemas changed. Use after a CLI version upgrade or when `check` reports missing components.

### Diagnose

```bash
npx create-agentlink check
```

Outputs JSON with `ready`, `supabase_running`, `database` (extensions, queues, functions, secrets, api_schema), and `files`. Read-only — reports problems but does not fix them.

### Component info

```bash
npx create-agentlink info          # Summary list
npx create-agentlink info <name>   # Detail for one component
```

Shows type, summary, description, signature, and related components. Use to understand what a missing component does.

### Flags

| Flag | Effect |
|------|--------|
| `--no-skills` | Skip companion skill installation |
| `--nextjs` | Use Next.js instead of Vite for frontend |
| `--no-frontend` | Skip frontend scaffolding (backend only) |
| `--no-launch` | Skip launching Claude Code after scaffold |
| `-y, --yes` | Auto-confirm all prompts |
| `--force-update` | Force update even if project is up to date |
| `--cloud` | Use Supabase cloud instead of local Docker |
| `--debug` | Write detailed log to `agentlink-debug.log` |
| `--plugin-dir <path>` | Use a local plugin directory instead of marketplace |

## Migration System

The CLI uses a **two-tier migration system** because `supabase db diff` cannot capture everything.

### Tier 1: Template migrations (hand-crafted)

Pre-written SQL files that the CLI writes to `supabase/migrations/` with timestamped filenames. Two categories:

**Pre-start migrations** — Infrastructure that must exist before `supabase start`:
- Extensions (`pg_net`, `pg_cron`, `pgmq`)
- Schema creation (`api` schema + grants)

These are applied automatically by `supabase start` when it runs migrations.

**Post-setup migrations** — Things that depend on application SQL and that `db diff` can't capture:
- Queue creation (`pgmq.create()` uses DO blocks — invisible to diff)
- Auth triggers (operate on `auth.users` — outside diff's scope)

Post-setup migrations are written *after* `db diff` runs and marked as applied via `supabase migration repair --status applied` (`--local` for local, `--linked` for cloud).

### Tier 2: Application migrations (generated by `pgdelta`)

**Prerequisite:** `pgdelta` is bundled with the CLI — no separate install needed.

```bash
# Local or Cloud (DB URL auto-resolved from .env.local)
npx create-agentlink@latest db apply
npx create-agentlink@latest db migrate descriptive_name

# Cloud only: push after generating
supabase db push
```

Captures everything in `public` and `api` schemas: tables, functions, indexes, policies, triggers. Uses `pgdelta` instead of `supabase db diff` to avoid alphabetical ordering issues with cross-file FK references. This is the standard migration workflow described in the `database` skill.

### Scaffold flow (local)

```
1. Write pre-start migrations (extensions, schemas)
2. supabase start              ← applies pre-start migrations
3. runSQL (schema files via psql) ← creates tables, functions, policies
4. db diff -f agentlink_setup  ← captures application SQL as migration
5. Write post-setup migrations  ← queues, auth triggers
6. migration repair            ← marks post-setup as applied
```

### Scaffold flow (cloud)

```
1. Write pre-start migrations (extensions, schemas)
2. Create cloud project + wait for ready
3. supabase link               ← links to cloud project
4. db push                     ← applies pre-start migrations
5. Write application migration ← schema SQL as a migration file
6. Write post-setup migrations ← queues, auth triggers
7. db push                     ← applies all migrations to cloud
8. Store vault secrets + set edge function secrets
9. Deploy edge functions
```

### Update flow

```
1. Write new template migrations (if any)
2. migration repair            ← mark new templates as applied
3. runSQL (schema files via psql) ← apply latest SQL
4. db diff -f update_name      ← capture any schema changes
```

## Schema Files vs Migrations

**Schema files** (`supabase/schemas/`) are the source of truth for application SQL. They use idempotent patterns (`CREATE OR REPLACE`, `IF NOT EXISTS`) and are applied via `psql`. The agent writes and modifies these during development.

**Migrations** (`supabase/migrations/`) are the deployment record for production. They are generated by `db diff`, which compares the live database against a shadow database built from existing migrations + `schema_paths`.

The `schema_paths` setting in `config.toml` tells `db diff` where to find schema files:

```toml
[db.migrations]
schema_paths = ["./schemas/_schemas.sql", "./schemas/_extensions.sql", "./schemas/**/*.sql"]
```

`db diff` bridges the two: it reads schema files to know the desired state, replays migrations on a shadow DB for the current state, and outputs the delta.

## When to Fix Manually

The CLI handles most cases, but if it fails or produces incorrect results, the agent can intervene:

1. **Write a migration file directly** — Create `supabase/migrations/<timestamp>_<name>.sql` with the correct SQL
2. **Mark it as applied** — `supabase migration repair <version> --status applied --local` (or `--linked` for cloud)
3. **Apply SQL via psql** — `psql <db_url> -c "SQL"` or pipe a schema file
4. **Fix a broken migration** — Edit the file in `supabase/migrations/`, then repair

Always prefer the CLI (`--force-update`) first. Only fix manually when the CLI can't handle the situation.
